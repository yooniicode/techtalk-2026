리허설 대본

발표 데모를 처음부터 끝까지 순서대로 정리한 대본. Discord에서 커맨드를 치며 진행한다.

시작 전 점검

  • Docker Desktop 실행 중, docker compose ps에서 opencode/bot 둘 다 Up
  • Discord 앱 Ctrl+R로 새로고침 (커맨드 16개 최신화)
  • 데모용 채널 하나 정해두기
  • 발표 화면에 Discord + (선택) docker compose logs -f 로그 띄우기

한 가지 기억할 점: 채널에서 커맨드를 처음 치면 봇이 주제별 스레드를 자동으로 만들고 거기서 작업한다. 같은 스레드 안에서는 대화 맥락(세션)이 이어진다. 주제가 바뀌면 새 채널에서 시작해 새 스레드를 판다.

0. 연결 확인

/ping

Bot: ✅ | opencode: ✅ 연결됨이 나오면 시작.

1. 코드 작업 — 레포 붙이고 이슈 처리

1-1. 레포 clone (autocomplete)

/repo

파라미터 자리에서 타이핑하면 내 GitHub 레포 목록이 드롭다운으로 뜬다. 원하는 레포를 고른다. 새 스레드가 생기고, 그 안에서 clone 진행 상황이 실시간으로 표시된다.

데모는 개인 레포(yooniicode/techtalk-2026)로 하는 게 안전하다. 이슈 생성·PR을 실제로 만들기 때문에 팀 레포는 부담이 된다.

1-2. 이슈 만들기 (스마트 생성)

만들어진 스레드 안에서 한 줄만 던진다:

/newissue 로그인 시 토큰 만료 처리가 안 되는 것 같아

승인 버튼을 누르면 에이전트가 레포 코드를 살펴보고 제목·배경·관련 파일 경로·재현 단계· 완료 조건을 갖춘 이슈를 만든 뒤 링크를 준다. 한 줄이 제대로 된 이슈가 된다.

1-3. 이슈 읽고 브랜치 만들기

방금 만든 이슈 번호로:

/issue 1 이슈 내용대로 구현 방향을 설명하고 브랜치 만들어줘

gh issue view로 이슈를 읽고, 요약한 뒤 연관 브랜치를 만든다.

1-4. 세션이 이어지는지 확인

같은 스레드에서:

/ask 방금 만든 브랜치 이름이 뭐였지?

이전 맥락을 기억하고 답하면 세션 유지가 되는 것.

1-5. 변경 검토

/diff

현재 브랜치에서 뭐가 바뀌었는지 파일별로 요약. 커밋 전에 사람이 확인하는 단계.

2. 안전장치 — 승인과 권한

2-1. PR 생성 (실행 전 승인 버튼)

/pr

먼저 ✅ 승인 / ❌ 거부 버튼이 뜬다. 요청한 사람만 누를 수 있다. 승인하면 push하고 PR을 만든다.

push 단계에서 opencode가 git push를 위험 명령으로 보고 권한을 다시 묻는다 — 이때 한 번만 허용 / 항상 허용 / 거부 3버튼이 뜬다. 선택하면 이어서 진행하고, 완료되면 PR 링크를 준다.

2-2. 코드리뷰

/review

PR 번호를 주면 그 PR을, 안 주면 현재 브랜치를 리뷰한다. 버그·보안·개선점을 파일:라인과 함께 짚는다.

3. 모델 라우팅 — 비싼 모델은 계획, 싼 모델은 실행

3-1. 계획 (Sonnet)

/plan 회원 탈퇴 API를 추가하는 방법

Sonnet 4.6이 단계별 계획만 세운다. 이 에이전트는 파일 수정 권한이 없어서 코드를 건드리지 않는다.

3-2. 실행 (Haiku)

/execute

바로 앞 /plan 결과를 Haiku 4.5가 실행한다. 실행 전 승인 버튼이 뜬다.

3-3. 방향 수정 (steering)

/steer 방금 거 말고 소프트 딜리트 방식으로 다시 해줘

같은 스레드 세션을 이어받아 방향을 튼다.

4. 멀티에이전트 — oh-my-opencode

/omo 이 레포에 헬스체크 API를 추가해줘

오케스트레이터가 작업을 전문 에이전트들에게 분담한다: explorer가 코드를 탐색하고, oracle이 설계하고, fixer가 구현한다. 역할마다 다른 Bedrock 모델을 쓴다(총괄은 Sonnet, 실무는 Haiku).

5. 제2의 뇌 — 옵시디언 볼트

5-1. 볼트에 기록

/note 오늘 배운 Docker 볼륨 개념 정리해서 저장해줘

옵시디언 볼트에 노트를 만들거나 갱신한다. 기존 폴더 구조와 위키링크 규칙을 따른다.

5-2. 볼트 내용으로 블로그 발행

새 채널에서 시작하는 걸 권장 (별도 스레드):

/blog AWS RDS 관련해서 볼트에 있는 내용으로 글 써줘

승인 버튼 → 볼트에서 관련 노트를 읽고 → Jekyll 포스트를 작성 → 커밋하고 push한다.

6. 비용 공개

/cost

이 스레드에서 지금까지 쓴 토큰과 비용을 보여준다. 계획(Sonnet)과 실행(Haiku)의 비용 차이를 얘기하는 마무리로 쓴다.

검증 우선순위

문제가 생기면 아래 순서로 확인한다.

  1. 1번 전체 — 기본 파이프라인. 제일 중요.
  2. 2-1 권한 버튼 — SSE로 권한 요청을 받아 버튼을 띄우고 다시 응답하는 흐름. 이번 구현의 최대 난이도.
  3. 1-3 세션 유지 — 같은 스레드에서 맥락을 기억하는지.
  4. 4번 /omo — 멀티에이전트가 Bedrock으로 도는지.

자주 막히는 곳

  • 커맨드가 안 보임 → Discord Ctrl+R. 그래도 안 되면 봇이 실행 중인지 docker compose logs bot 확인.
  • /pr에서 권한 에러 → GitHub 토큰이 그 레포에 push/PR 권한이 있어야 한다. 개인 레포로 데모하는 게 안전.
  • 응답이 느림 → clone이나 파일 여러 개 읽는 작업은 시간이 걸린다. 스레드의 진행 표시로 지금 뭐 하는지 확인.
  • 볼트 수정이 걱정 → 볼트는 git 레포라 문제 생기면 git checkout으로 되돌릴 수 있다. /ask는 볼트를 읽기만 하고, /note만 쓴다.
